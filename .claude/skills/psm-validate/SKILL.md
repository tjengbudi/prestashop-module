# psm-validate

## Overview

Bertindak sebagai validator module PrestaShop yang teliti dan jujur: operator memegang module, skill memegang aturan kompatibilitas lintas versi dan prosedur ujinya. Hasilkan **vonis berbasis bukti** apakah sebuah module sehat di PrestaShop **1.7.x, 8.x, dan 9.x sekaligus** — empat lapis: aturan deterministik yang diketahui pasti (skrip), perilaku nyata terhadap PrestaShop core asli di Docker flashlight, review adversarial e-commerce (judgment), dan uji perilaku browser (E2E) terhadap core yang benar-benar boot. Konsumen hasil: operator (perbaiki sebelum rilis) dan workflow lain (psm-cross-version, psm-develop, psm-scaffold) yang memanggil skill ini sebagai gerbang mutu — jadi output harus JSON terstruktur yang dapat dibaca mesin, dengan ringkasan yang bisa ditindaklanjuti manusia. Tidak pernah meloloskan module yang masih memakai dependency terlarang PS9, hook yang dihapus, atau tanpa `ps_versions_compliancy`.

## Resolution rules

- Bare paths dan `{skill-root}` (mis. `assets/ps-rules.json`) resolve dari direktori instal skill ini.
- `{project-root}` → direktori kerja project.

## On Activation

1. Muat config resolved via `uv run {project-root}/.claude/skills/psm-setup/scripts/resolve-psm-config.py --project-root {project-root}` dan baca apa adanya — default kanonik sudah diterapkan resolver (jangan parse `config.yaml` sendiri). Resolver absen (disalin tanpa psm-setup)? Lanjut dengan default kanonik skrip dan catat bahwa default dipakai. Setiap lapis menerima setelannya sebagai flag di titik pakainya; URL yang di-drive Playwright memakai ulang `psm_flashlight_ps_domain`.
2. Tentukan module yang divalidasi (path folder) dari permintaan user; nama telanjang di-resolve terhadap `psm_modules_dir`, bila tetap ambigu tanyakan. Interaktif: sebelum lapis jalan, tawarkan cakupan uji — versi dari `psm_target_versions`, browser Lapis 4 dari `psm_e2e_browsers`; default semua yang terkonfigurasi, dan pilihan itu dipakai seragam di semua lapis (`--versions`, `--browsers`).
3. **Augment aturan bila knowledge base ada.** Bila `{project-root}/_bmad/psm/memory/tech/validator-rules.md` atau `tech/flashlight.md` ada, baca: aturan pindai tambahan (JSON ber-skema `assets/ps-rules.json`) diteruskan sebagai `--extra-rules <file>` di Lapis 1, tag flashlight tambahan sebagai `--extra-tag-map` di Lapis 2/4 — dua flag yang MENAMBAH; `--rules` dan `--tag-map` MENGGANTI, jadi jangan pakai keduanya untuk tambahan. Jangan terapkan aturan dengan tangan. Bila belum ada, lanjut tanpanya — aturan inti sudah di-embed.

## Validate

Jalankan empat lapis pengujian lalu satukan jadi vonis — lapis murah dan deterministik dulu supaya run mahal tak diulang. Tulis output tiap lapis ke `<psm_reports_dir>/<module>-<lapis>.json` (`static`/`flashlight`/`adversarial`/`e2e`). Untuk melanjutkan run yang terputus tanpa memvonis atas bukti basi, jalankan `uv run scripts/ps-plan-layers.py <module-path> --reports-dir <psm_reports_dir> --versions <target>` lebih dulu dan jalankan lapis yang ditandainya `rerun` — jangan menilai kesegaran file dengan tangan.

**Lapis 1 — pindai statis lintas versi (selalu).** Jalankan `uv run scripts/ps-static-scan.py <module-path> --versions <target>` (lihat `--help`). Skrip mencocokkan ruleset lintas-versi `assets/ps-rules.json` ke source module dan mengeluarkan temuan JSON per versi dengan `pass`/`errors`/`warnings`. Ini sumber kebenaran untuk aturan yang diketahui pasti — jangan menilai ulang temuannya dengan tangan. `main_file_found: false` berarti folder itu bukan module PrestaShop — **berhenti**: lapis lain mahal dan tak ada yang bisa dinilai, dan temuan compliancy atas file yang tak ada itu menyesatkan. Interaktif: sebutkan + tawarkan kandidat dari `psm_modules_dir`; headless: `blocked`.

**Lapis 2 — uji di flashlight per versi (bila Docker ada).** Jalankan `uv run scripts/ps-flashlight-run.py <module-path> --versions <target>` dan teruskan setelan config sebagai flag (`--tag-map` dari `psm_flashlight_tag_map`, `--db-image`, `--ps-domain`, `--orchestrator`, `--startup-timeout`) supaya sekali panggil langsung jalan (lihat `--help`). Flashlight butuh DB terpisah (image-nya web-tier saja) — skrip mengurus pasangannya dan menjalankan **phpstan** terhadap core asli (neon module ada = konklusif; auto-generate = advisory, tak memblok). Docker/compose absen → `status: skipped`; image belum ada lokal → `skipped_image`; DB/boot gagal → tak konklusif. Image flashlight besar: di sesi interaktif konfirmasi dulu, lalu ulangi dengan `--allow-image-pull`.

**Lapis 3 — review adversarial e-commerce (judgment).** Delegasikan ke satu subagent reviewer supaya source module tak membebani konteks orkestrasi: beri path module, versi target (juga saat cakupan dipersempit), dan **selalu** `references/adversarial-lens.md` — sikap, lensa inti, dan bentuk kembalian yang ditegakkan agregat. Bila `{project-root}/_bmad/psm/memory/ecommerce/adversarial-checks.md` ada, sertakan sebagai lensa **tambahan**; ia menambah pertanyaan, tak pernah mengganti bentuk kembalian. Tulis hasilnya ke file lapis adversarial (konvensi di atas) supaya skrip agregat memperlakukannya setara lapis lain. Bila subagent tak tersedia (skill ini sendiri sedang berjalan sebagai subagent), baca dan review sendiri dengan kontrak yang sama.

**Lapis 4 — uji perilaku browser E2E (bila `psm_e2e_enabled` ≠ false dan Docker + browser Playwright ada).** Prasyarat sekali: `playwright install <engine dari psm_e2e_browsers>` (setup mesin baru + loop TDD E2E: `references/e2e-quickstart.md`). Jalankan `uv run scripts/ps-e2e-run.py <module-path> --versions <target> --browsers <browser>` dengan setelan flashlight sama seperti Lapis 2. Skrip men-drive PrestaShop core asli lewat Playwright — smoke universal (shop tak rusak dengan module aktif) plus skenario authored module di `<module-path>/tests/e2e/*.json` — dan memvonis perilaku; lihat `--help` untuk format spec, jangan nilai ulang outputnya. Verifikasi visual (interaktif; lewati di headless — tak ada peninjau): sertakan `--screenshot-dir <psm_reports_dir>/e2e-shots`, lalu tinjau screenshot kunci dengan penglihatanmu — assertion buta pada layout ambruk atau blok kosong. Cacat visual yang kamu yakini ditulis ke file lapis adversarial sebagai `error` **sebelum** agregat jalan — itu kanal sah judgment model ke vonis; sisanya catatan satu baris. Alat lain (`--headed`, error console): `references/e2e-quickstart.md`. Degrade jujur seperti Lapis 2 (plus `skipped_browser` bila browser absen).

## Vonis dan output

Serahkan penggabungan ke skrip — jangan rakit vonis dengan tangan atau menilai ulang/menimpa field yang dikeluarkannya. Jalankan `uv run scripts/ps-aggregate.py` dengan file lapis dari konvensi di atas (lewati flag lapis yang tak dijalankan; lihat `--help`) dan `-o <psm_reports_dir>/<module>-<timestamp>.json`. Exit 2 dari skrip manapun = input melanggar skema (file adversarial, aturan tambahan) — perbaiki filenya lalu ulangi, jangan tafsirkan sebagai vonis.

Untuk `{user_name}` (kosong → "operator"), ringkas dalam percakapan (bahasa: `{communication_language}`) dari JSON itu: per versi lolos/gagal, error yang memblok beserta fix-nya, dan warning yang sebaiknya ditangani. Aturan jujur, sekali untuk semua lapis — sebut eksplisit yang tak konklusif (`flashlight_conclusive`/`e2e_conclusive` false, `inconclusive_note`), lapis yang tak jalan (`layers_run`), cakupan versi/browser yang dipersempit, spec E2E authored yang dilewati (`e2e_scenario_notes` — uji yang ditulis tapi tak jalan), dan E2E yang cuma smoke (`e2e_smoke_only`: shop terbukti tak rusak, perilaku module belum teruji). Untuk dua yang terakhir tawarkan perbaikannya: perbaiki/tulis skenario di `<module>/tests/e2e/`. `pass` buta konklusivitas — `ready` yang menjawab siap-rilis. Sebutkan juga folder screenshot (`e2e_screenshot_dir`) bila ada. Vonis tak pernah mengklaim coverage yang tak diuji. Bila vonis keseluruhan gagal, tutup dengan satu tawaran langkah lanjut: serahkan error pemblokir ke psm-develop untuk diperbaiki, atau validasi ulang setelah dipatch.

## Mode headless

Saat dipanggil workflow lain atau dengan `--headless`, lewati pertanyaan klarifikasi: ambil module-path, versi & browser dari argumen/konteks, jalankan lapis yang tersedia, tulis JSON. Jalankan Lapis 2 & Lapis 4 **tanpa** `--allow-image-pull` — jangan pernah tarik image tanpa manusia (pemanggil yang memang mau menarik meneruskannya eksplisit); binary browser harus sudah di-`playwright install` sebelumnya. Catat asumsi yang diambil (module-path di-infer, default resolver dipakai, review sendiri pengganti subagent) ke `<psm_reports_dir>/<module>-validate.memlog.md` via `uv run {project-root}/_bmad/scripts/memlog.py` — `init --path <file> --field skill=psm-validate` dulu bila file itu belum ada (`append` ke file absen gagal), lalu `append --path <file> --type assumption --text <asumsi>`.

Kembalikan objek JSON kecil, bukan prosa: `status` (`complete`|`blocked`), path laporan, path memlog, `ready`, `pass`, dan konklusivitas (`flashlight_conclusive`, `e2e_conclusive`, `e2e_smoke_only`, `layers_run`). `blocked` + `reason` satu baris bila skill tak bisa menilai sama sekali: module-path tak resolve, `main_file_found: false` (bukan module PrestaShop), Lapis 1 tak menghasilkan JSON, atau exit 2 skema. Gerbang CI membaca **`ready`**: lolos + keempat lapis tuntas (default `--require`). Runner tanpa Docker/browser memang menghasilkan `ready: false` — itu jawaban jujur "mesin ini tak bisa membuktikan siap rilis", bukan kegagalan module; `pass` tetap ada untuk membaca temuan yang benar-benar ditemukan. Sengaja menggating lebih sempit? Nyatakan (mis. `--require static,adversarial`) — pilihan itu terekam di `required_layers`.
