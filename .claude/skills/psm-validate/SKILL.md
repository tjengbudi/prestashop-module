# psm-validate

## Overview

Bertindak sebagai validator module PrestaShop yang teliti dan jujur: operator memegang module, skill memegang aturan kompatibilitas lintas versi dan prosedur ujinya. Hasilkan **vonis berbasis bukti** apakah sebuah module sehat di PrestaShop **1.7.x, 8.x, dan 9.x sekaligus** ‚Äî empat lapis: aturan deterministik yang diketahui pasti (skrip), perilaku nyata terhadap PrestaShop core asli di Docker flashlight, review adversarial e-commerce (judgment), dan uji perilaku browser (E2E) terhadap core yang benar-benar boot. Konsumen hasil: operator (perbaiki sebelum rilis) dan workflow lain (psm-cross-version, psm-develop, psm-scaffold) yang memanggil skill ini sebagai gerbang mutu ‚Äî jadi output harus JSON terstruktur yang dapat dibaca mesin, dengan ringkasan yang bisa ditindaklanjuti manusia.

## Resolution rules

- Bare paths dan `{skill-root}` (mis. `assets/ps-rules.json`) resolve dari direktori instal skill ini.
- `{project-root}` ‚Üí direktori kerja project.

## On Activation

1. Muat config resolved via `uv run {project-root}/.claude/skills/psm-setup/scripts/resolve-psm-config.py --project-root {project-root}` dan baca apa adanya ‚Äî default kanonik sudah diterapkan resolver (jangan parse `config.yaml` sendiri). Resolver absen (disalin tanpa psm-setup)? Lanjut dengan default kanonik skrip dan catat bahwa default dipakai. Setiap lapis menerima setelannya sebagai flag di titik pakainya; URL yang di-drive Playwright memakai ulang `psm_flashlight_ps_domain`.
2. Tentukan module yang divalidasi (path folder) dari permintaan user; nama telanjang di-resolve terhadap `psm_modules_dir`, bila tetap ambigu tanyakan. Interaktif: sebelum lapis jalan, tawarkan cakupan uji ‚Äî versi dari `psm_target_versions`, browser Lapis 4 dari `psm_e2e_browsers`; default semua yang terkonfigurasi, dan pilihan itu dipakai seragam di semua lapis (`--versions`, `--browsers`).
3. **Augment aturan (opsional).** Aturan pindai di luar `assets/ps-rules.json` dan tag di luar `psm_flashlight_tag_map` hanya masuk lewat artefak eksplisit ‚Äî `--extra-rules <file.json>` (Lapis 1, menambah aturan) dan `--extra-tag-map <versi=tag>` (Lapis 2/4, menambal tag: versi yang sama ‚Üí yang tambahan menang); `--rules`/`--tag-map` MENGGANTI seluruhnya. Jangan terjemahkan prosa jadi aturan dengan tangan ‚Äî tanpa artefak itu, aturan inti sudah di-embed.

## Validate

Jalankan empat lapis pengujian lalu satukan jadi vonis ‚Äî lapis murah dan deterministik dulu supaya run mahal tak diulang. Tiap skrip di bawah memiliki vonisnya sendiri: baca apa adanya, jangan nilai ulang, timpa, atau rakit ulang dengan tangan. Tulis output tiap lapis ke `<psm_reports_dir>/<module>-<lapis>.json` (`static`/`flashlight`/`adversarial`/`e2e`). Untuk melanjutkan run yang terputus tanpa memvonis atas bukti basi, jalankan `uv run scripts/ps-plan-layers.py <module-path> --reports-dir <psm_reports_dir> --versions <target>` lebih dulu (teruskan `--extra-rules` yang sama bila dipakai) dan jalankan lapis yang ditandainya `rerun` (perbaiki spec di `e2e_scenario_notes` sebelum Lapis 4 boot).

**Lapis 1 ‚Äî pindai statis lintas versi (selalu).** Jalankan `uv run scripts/ps-static-scan.py <module-path> --versions <target>` (lihat `--help`). Skrip mencocokkan ruleset lintas-versi `assets/ps-rules.json` ke source module dan mengeluarkan temuan JSON per versi dengan `pass`/`errors`/`warnings`. Ini sumber kebenaran untuk aturan yang diketahui pasti. `main_file_reason: no_php_at_root` berarti folder itu bukan module PrestaShop ‚Äî **berhenti**: lapis lain mahal dan tak ada yang bisa dinilai, dan temuan compliancy atas file yang tak ada itu menyesatkan. Interaktif: sebutkan + tawarkan kandidat dari `psm_modules_dir`; headless: `blocked`. `ambiguous_main_file` bukan alasan berhenti: `main_file_candidates` memuat `extends_module` ‚Äî sebut file utama yang tepat atau folder yang perlu dinamai ulang.

**Lapis 2 ‚Äî uji di flashlight per versi (bila Docker ada).** Jalankan `uv run scripts/ps-flashlight-run.py <module-path> --versions <target>` dan teruskan setelan config sebagai flag (`--tag-map` dari `psm_flashlight_tag_map`, `--db-image`, `--ps-domain`, `--orchestrator`, `--startup-timeout`) supaya sekali panggil langsung jalan (lihat `--help`). Skrip mengurus DB, phpstan, dan degrade sendiri lalu menandainya di output. Image flashlight besar: di sesi interaktif konfirmasi dulu, lalu ulangi dengan `--allow-image-pull`. Run yang di-kill paksa meninggalkan container + network `psm-fl` ‚Üí run berikut gagal bind; bersihkan: `docker ps -aq --filter name=psm-fl | xargs -r docker rm -f && docker network ls --filter name=psm-fl -q | xargs -r docker network rm`.

**Lapis 3 ‚Äî review adversarial e-commerce (judgment).** Delegasikan ke satu subagent reviewer supaya source module tak membebani konteks orkestrasi: beri path module, versi target (juga saat cakupan dipersempit), dan **selalu** `references/adversarial-lens.md` ‚Äî sikap, lensa inti, dan bentuk kembalian yang ditegakkan agregat. Bila `{project-root}/_bmad/psm/memory/ecommerce/adversarial-checks.md` ada, sertakan sebagai lensa **tambahan**; ia menambah pertanyaan, tak pernah mengganti bentuk kembalian. Tulis hasilnya ke file lapis adversarial (konvensi di atas) supaya skrip agregat memperlakukannya setara lapis lain. Bila subagent tak tersedia (skill ini sendiri sedang berjalan sebagai subagent), baca dan review sendiri dengan kontrak yang sama.

**Lapis 4 ‚Äî uji perilaku browser E2E (bila `psm_e2e_enabled` ‚âÝ false dan Docker + browser Playwright ada).** Prasyarat sekali (browser Playwright, setup mesin baru + loop TDD E2E): `references/e2e-quickstart.md`. Jalankan `uv run scripts/ps-e2e-run.py <module-path> --versions <target> --browsers <browser>` dengan setelan flashlight sama seperti Lapis 2 (format spec & degrade: `--help`). Verifikasi visual (interaktif; lewati di headless ‚Äî tak ada peninjau): sertakan `--screenshot-dir <psm_reports_dir>/e2e-shots`, lalu tinjau screenshot kunci dengan penglihatanmu ‚Äî assertion buta pada layout ambruk atau blok kosong. Cacat visual yang kamu yakini **memblok**: tulis ke file lapis adversarial sebelum agregat jalan ‚Äî satu-satunya kanal judgment model ke vonis; sisanya catatan satu baris. Cara menulisnya (dan alat lain: `--headed`, error console): `references/e2e-quickstart.md`.

## Vonis dan output

Serahkan penggabungan ke skrip. Jalankan `uv run scripts/ps-aggregate.py` dengan file lapis dari konvensi di atas ‚Äî sertakan tiap lapis yang **punya file**, termasuk yang plan tandai `reuse`; lewati hanya lapis tanpa file (lihat `--help`) ‚Äî dan `-o <psm_reports_dir>/<module>-<timestamp>.json`. Exit 2 dari skrip manapun = error input (skema file adversarial/aturan tambahan, path salah, versi target tak dipindai Lapis 1, lapis `--require` tak dikenal) ‚Äî baca stderr, perbaiki inputnya lalu ulangi; jangan tafsirkan sebagai vonis.

Untuk `{user_name}` (kosong ‚Üí "operator"), ringkas dalam percakapan (bahasa: `{communication_language}`) dari JSON itu: per versi lolos/gagal, error yang memblok beserta fix-nya, dan warning yang sebaiknya ditangani. Aturan jujur, sekali untuk semua lapis ‚Äî sebut eksplisit yang tak konklusif (`flashlight_conclusive`/`e2e_conclusive` false, `inconclusive_note`; `advisory_note` dilaporkan tapi tak menggerbang), lapis yang tak jalan (`layers_run`), cakupan versi/browser yang dipersempit (`e2e_browsers`), spec E2E authored yang dilewati (`e2e_scenario_notes`), dan E2E yang cuma smoke (`e2e_smoke_only`: shop terbukti tak rusak, perilaku module belum teruji ‚Äî ini **menjatuhkan `ready`**). Untuk dua yang terakhir tawarkan perbaikannya: perbaiki/tulis skenario di `<module>/tests/e2e/` (format + pola BO lintas-versi: `references/e2e-quickstart.md`); sadar memilih rilis tanpa uji perilaku? sempitkan gerbang lewat `--require`, jangan abaikan `ready`. `pass` buta konklusivitas ‚Äî `ready` yang menjawab siap-rilis. Sebutkan juga folder screenshot (`e2e_screenshot_dir`) bila ada. Bila vonis keseluruhan gagal, tutup dengan satu tawaran langkah lanjut: serahkan error pemblokir ke psm-develop untuk diperbaiki, atau validasi ulang setelah dipatch.

## Mode headless

Saat dipanggil workflow lain atau dengan `--headless`, lewati pertanyaan klarifikasi: ambil module-path, versi & browser dari argumen/konteks, jalankan lapis yang tersedia, tulis JSON. Jalankan Lapis 2 & Lapis 4 **tanpa** `--allow-image-pull` ‚Äî jangan pernah tarik image tanpa manusia (pemanggil yang memang mau menarik meneruskannya eksplisit); binary browser harus sudah di-`playwright install` sebelumnya. Catat asumsi yang diambil (module-path di-infer, default resolver dipakai, review sendiri pengganti subagent) ke `<psm_reports_dir>/<module>-validate.memlog.md` via `uv run {project-root}/_bmad/scripts/memlog.py` ‚Äî `init --path <file> --field skill=psm-validate` dulu bila file itu belum ada (`append` ke file absen gagal), lalu `append --path <file> --type assumption --text <asumsi>`.

Kembalikan objek JSON kecil, bukan prosa: `status` (`complete`|`blocked`), path laporan, path memlog, `ready`, `pass`, dan konklusivitas (`flashlight_conclusive`, `e2e_conclusive`, `layers_run`, plus `e2e_smoke_only` bila E2E jalan). `blocked` + `reason` satu baris bila skill tak bisa menilai sama sekali: module-path tak resolve, `main_file_reason: no_php_at_root` (bukan module PrestaShop), Lapis 1 tak menghasilkan JSON, atau exit 2 error input. Gerbang CI membaca **`ready`**: lolos + lapis yang di-`--require` tuntas (default keempatnya). Runner tanpa Docker/browser menghasilkan `ready: false` ‚Äî jujur, bukan cacat module; `pass` tetap menampung temuan nyata. Menggating lebih sempit: nyatakan di `--require`.
